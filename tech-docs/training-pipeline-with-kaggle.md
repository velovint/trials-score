# Automated ML Pipeline: Android-Native OpenCV & LiteRT

This document defines the architecture and testing gates for the card scanning pipeline, ensuring **consistency** between training data preparation and on-device inference.

> **Architecture Note:** This pipeline uses Android-only modules (not KMP) because LiteRT and OpenCV expose fundamentally different APIs between Android and JVM platforms. Attempting to share code would require extensive abstraction layers that add complexity without meaningful benefit.

## 🏗️ Project Modules

* **`:shared-cv`**: Android library module. Contains OpenCV grid extraction logic (`CardImagePreprocessor`).
* **`:ml-inference`**: Android library module. Contains the **LiteRT (TFLite) Classifier** and model inference.
* **`:data-prep-tool`**: Android "Worker" module. Uses `:shared-cv` within **instrumented tests** to create training datasets from raw images.
* **`:app`**: Android Application. Coordinates CameraX, `:shared-cv`, and `:ml-inference`.

**Module Dependencies:**
```
:app → :ml-inference → :shared-cv
:app → :shared-cv (direct)
:data-prep-tool → :shared-cv
```

## 🗂️ Kaggle Integration Points

The pipeline integrates with Kaggle via three datasets plus a training notebook:

1. **Raw Images Dataset** (`<user>/trials-score-raw-images`) - Source score card images, updated occasionally
2. **Training Data Dataset** (`<user>/trials-score-training-data`) - Processed 640x66 row images, generated by pipeline
3. **Model Dataset** (`<user>/trials-score-model`) - Production `.tflite` file, updated after successful training
4. **Training Notebook** (`<user>/trials-score-trainer`) - Executes model training, triggered via Notebooks API

**Model Versioning:** Models are tagged as "latest" in Kaggle after successful validation. Workflow 1 always downloads the latest-tagged model. Optional: `ml-inference/MODEL_VERSION.txt` can track the specific version for reference (e.g., "1.3.0" or "latest"), but the model binary itself is never committed to git.

## 🔄 Two Workflows

### Workflow 1: Daily Development Build
**Purpose:** Build the application using the current production model.

**Entry Point:** `./gradlew :app:assembleDebug`

**Behavior:**
* Downloads "latest" tagged model from Kaggle Model Dataset (if not cached)
* Caches model using Kaggle dataset version/hash - reuses until Kaggle model updates
* All three test tiers run and must pass (validates current CV logic + model + integration)

### Workflow 2: Model Update Pipeline
**Purpose:** Generate and validate a new production model from raw training images.

**Entry Point:** `./gradlew buildProductionModel` (manual invocation)

**Steps:**
1. Download raw score card images from Kaggle (cached, checks dataset hash)
2. Run `:data-prep-tool:connectedAndroidTest` to preprocess images into training data (requires device/emulator)
3. Upload processed training data to Kaggle Training Data Dataset
4. Trigger Kaggle Notebook execution and wait for completion
5. Download new model from Kaggle Model Dataset
6. Run `connectedCheck` - all instrumented tests must pass to gate promotion

**After Success:** Developer tags model as "latest" in Kaggle Model Dataset (via Kaggle UI or API). Next Workflow 1 build automatically picks up the new model. Optionally update `MODEL_VERSION.txt` for reference tracking.

**Caching:** Gradle build cache handles training data (invalidated by raw image or `:shared-cv` code changes).

## 🧪 Testing Strategy

All validation runs as Android instrumented tests on emulator/device. This eliminates cross-platform inconsistencies at the cost of slower feedback loops.

### Tier 1: CV Logic Validation
* **Module:** `:shared-cv`
* **Action:** Instrumented tests with sample score card images validate Hough Transform, grid extraction, and preprocessing accuracy.
* **Workflow 1:** Validates current CV logic still works correctly
* **Workflow 2:** Must pass before training (broken CV = bad training data)

### Tier 2: Model Validation Gate
* **Module:** `:ml-inference`
* **Action:** Instrumented tests load the `.tflite` model and run it against 100 "Golden" images.
* **Workflow 1:** Verifies current model achieves ≥95% accuracy on device
* **Workflow 2:** Must pass before promotion - if accuracy < 95%, model NOT tagged as "latest"

### Tier 3: End-to-End Integration
* **Module:** `:app`
* **Action:** Uses mock camera images to test the full UI flow on an emulator ("Camera → CV → ML" wiring).
* **Workflow 1:** Validates app integration works end-to-end
* **Workflow 2:** Must pass before promotion - ensures new model works in real app context

**All tests fail the build if they don't pass.** The difference between workflows is context: Workflow 1 validates the current state, Workflow 2 validates new model before tagging it as "latest" in Kaggle.

## ⚠️ Critical Gotchas

1.  **OpenCV Initialization:**
    * All modules use `OpenCVLoader.initDebug()` before any OpenCV operations.
    * Instrumented tests must initialize OpenCV in `@Before` setup methods.

2.  **The "Mat" Contract:**
    * Always pass `org.opencv.core.Mat` between modules. Convert `Bitmap` only at the entry/exit points to keep the core logic pure.

3.  **Hardware Acceleration:**
    * Both training data preparation and inference run on Android devices/emulators.
    * LiteRT may use **GPU/NNAPI delegates** depending on device capabilities.
    * *Solution:* Use consistent delegate configuration across data prep and inference to avoid floating-point drift.

4.  **Kaggle API Authentication:**
    * Requires `~/.kaggle/kaggle.json` with valid credentials
    * Format: `{"username":"<user>","key":"<api-key>"}`
    * Obtain from Kaggle Account Settings → API → Create New API Token

5.  **Test Orchestrator Required:**
    * All instrumented tests use `ANDROIDX_TEST_ORCHESTRATOR` with `clearPackageData: true`
    * Ensures clean state between test runs

6.  **CV ↔ Model Compatibility:**
    * Test-based enforcement: If CV preprocessing changes but tests pass, build proceeds (compatible change)
    * If tests fail after CV changes, retraining required
    * No strict version coupling - trust test suite to catch incompatibilities

7.  **Model Storage:**
    * Model binaries are NOT committed to git - they live in Kaggle Model Dataset
    * After Workflow 2 validation passes, tag model as "latest" in Kaggle
    * Workflow 1 always downloads the latest-tagged model
    * Cache invalidation happens automatically when Kaggle dataset version changes

## 🐘 Gradle Task Reference

```bash
# Workflow 1: Daily Development
./gradlew :app:assembleDebug              # Build with cached model
./gradlew check connectedCheck            # Run all tests (unit + instrumented)

# Workflow 2: Model Update (requires device/emulator)
./gradlew buildProductionModel            # Full pipeline with validation gates
```

**Task Dependencies:**

Pipeline tasks declare inputs/outputs, and Gradle automatically chains them:
- `downloadRawImages` outputs → `build/raw-images/`
- `:data-prep-tool:connectedAndroidTest` consumes raw images, outputs training data
- `uploadTrainingData` consumes training data
- `triggerKaggleNotebook` waits for upload, triggers training
- `downloadModel` waits for training completion, outputs model file
- `connectedCheck` consumes model file (via asset merging)

```groovy
task("buildProductionModel") {
    description = "Full ML pipeline: prep → train → validate"
    group = "ml-pipeline"

    // Just depend on validation - Gradle chains the rest via inputs/outputs
    dependsOn 'connectedCheck'
}
```
